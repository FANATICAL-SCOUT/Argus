A bar chart of the most commonly found services across all your scans. Bars are coloured by service type. What to look for:

New Scan

Configure and launch a live TCP connect-scan

Target Host

Enter an IPv4 address (e.g. 192.168.1.1) or a resolvable hostname (e.g. myserver.local). You must own or have explicit written permission to scan the target.

Port Range

Format: start-end (e.g. 1-1024) or a comma-separated list of specific ports (e.g. 80,443,8080). The scanner probes every listed TCP port.

Vulnerability Detection toggle

When On (default), pscan checks each open port against a built-in CVE database immediately after finding it. You see vulnerability counts in real time during the scan. Turn it Off when you only need port discovery — this can cut scan time in half on large port ranges since CVE matching adds processing per port.

Host Alive Check toggle

When On, pscan first sends an ICMP ping to the target before starting the port scan. If the host doesn't respond (offline, ICMP blocked, wrong IP), the scan aborts with a clear message. Turn it Off if you know the host is up but ICMP is firewalled — many corporate devices block ping but still have open TCP ports.

Live Scan View

After launching, the page switches to live view. You'll see:

Scan History

Browse, filter, and manage all past scans

What is Scan History?

Scan History is your technical database of every scan you have run. It is designed for you (the operator) to browse raw data, compare scans over time, delete old records, and export data for further analysis. Think of it as your scan log.

How it differs from Reports: History shows you the raw technical output — exact port numbers, service banners, CVE IDs, timestamps. Reports format the same scan into a narrative security assessment suitable for sharing with stakeholders.

Type in Search targets to filter by IP or hostname. Use the Export dropdown to download the current (filtered) view as CSV or JSON. Click any column header to sort.

  • The Export button respects your current filters — filter first, then export.

Reports

Formatted security assessments with remediation advice

What is a Report?

A Report is a formatted security assessment generated from one scan. Unlike the raw Scan History detail view, a report is designed to be shared — with a manager, client, or audit team. It includes a risk badge, narrative descriptions, and auto-generated remediation steps.

VulnerabilitiesCVE ID, name, confidence (Confirmed / Potential), and description
RecommendationsAuto-generated remediation steps based on services and vulns found
Scan MetadataTarget, port range, start/end timestamps, scan ID

Risk Levels

BadgeVuln CountWhat it means
Clean0No CVE matches — lowest risk
Low Risk1–2Minor issues, review recommended
Medium Risk3–5Notable findings, prioritise patching
High Risk6+Critical — immediate action needed

Vulnerability Confidence

  • Confirmed — banner matched a known vulnerable version string. High confidence; verify before patching.
  • Potential — service type matched a vulnerability category but version could not be confirmed from the banner. Always verify manually.

Print / Export as PDF

Click Print / PDF in the report header. The sidebar and controls hide automatically for a clean printout. Use your browser's "Save as PDF" option to create a shareable PDF file.

Tips
  • Potential findings can be false positives — banner grabbing is imperfect. Always verify before acting.
  • Use the Detail View button in the report to jump back to the raw scan data.
  • The Reports index lets you filter by target IP to see all historical reports for one host.
History vs Reports
  • History = raw data log for the operator (you)
  • Reports = formatted assessment for sharing with others

Network Topology

Discover and visualise hosts on your local network

What is Network Topology?

Network Topology discovers every live host on your subnet and draws them as an interactive graph. It answers "what devices are on this network?" before you decide which ones to scan in detail — a LAN census that tells you what exists and what type of device it is.

Discovery Tiers

pscan uses three discovery methods simultaneously and merges the results:

TierMethodRequiresWhat it finds
1ARP scan (Scapy)Admin + NpcapEvery device on the subnet — firewall-independent
2OS ARP table readNothing (always runs)Recently-seen devices cached by the OS; captures MAC addresses
3ICMP ping + TCP probeNothing (always runs)Any host that responds to ping or common TCP ports

A banner at the top of the canvas shows which tier is active. Run pscan as Administrator for Tier 1 (best coverage). Tier 2+3 always run regardless.

Device Fingerprinting

After discovery, pscan port-scans each host on 27 fingerprint ports and classifies it:

Device TypeDetection signals
Router / GatewayGateway IP position, or Cisco/Huawei MAC vendor
Windows PCRDP (3389), SMB (445/135)
Linux ServerSSH + HTTP/S
Web ServerHTTP/HTTPS only
DatabaseMySQL (3306) or MSSQL (1433)
PrinterIPP (631), JetDirect (9100), LPD (515)
VMware ESXiPort 902
SDN ControllerOpenFlow (6633/6653), ONOS/ODL REST (8181)

OS hint comes from port patterns first, then TTL value from the ping response (128 → Windows, 64 → Linux/macOS, 255 → Network Device). Vendor name comes from the MAC OUI prefix lookup.

Graph View

  • Node colour — device type (blue=router, indigo=Windows, green=Linux/server, teal=SDN, etc.)
  • Node border colour — risk level (green=clean, yellow=low, orange=medium, red=high)
  • Gateway node — diamond shape, larger; all edges point from gateway to hosts
  • This device — double border ring
  • Edge width — scales with target's open port count; thicker = more services
  • Edge colour — matches target's risk level
  • Click any node to open the detail panel (IP, MAC, vendor, OS hint, open ports)

SDN Controller Sidebar Panel

After every discovery, the sidebar shows a real-time analysis of your network:

  • Controller Detection — green dot if an OpenFlow/ONOS node was found; grey if standard L2 network
  • Network Segments — device type breakdown with host counts and colour codes
  • Derived Flow Policies — which services (SSH, Telnet, RDP, SMB…) are exposed across how many hosts; risky services flagged in red
  • Network Health Score — 0–100 score; loses points for high-risk hosts and dangerous open services (Telnet, FTP, VNC)
Tips
  • Run pscan as Administrator to enable Tier 1 ARP scanning — it finds devices that block ICMP (printers, IoT, firewalled hosts).
  • Start with /24 for a fast LAN sweep (~30 s). Only use /16 when you need corporate-wide coverage.
  • Use the Filter bar to find a specific IP or vendor instantly in large topologies.
  • Click any node → Full Port Scan to immediately deep-scan that host.
  • Use Export PNG to save the topology for documentation or incident reports.
Red flags
  • An unknown IP node = possible rogue device — investigate immediately.
  • Any host with a red border (high risk) should be prioritised for a full scan.
  • Health score below 50 = multiple dangerous services exposed — check Derived Flow Policies for specifics.
  • An SDN Controller detected on an unexpected IP = misconfigured or rogue controller.
Discovery mode banner
  • Green — ARP scan active; fullest coverage
  • Blue — ARP table + ping/TCP; good coverage, no Scapy needed
  • Yellow — ping/TCP only; may miss firewalled/sleeping devices

SDN Controller

OpenFlow 1.3 — software-defined network management console

What is SDN?

Software-Defined Networking (SDN) separates the network's control plane (decisions about where traffic goes) from the data plane (the physical forwarding of packets). Instead of configuring each switch individually, an SDN controller pushes flow rules to all managed devices centrally. OpenFlow 1.3 is the standard protocol used to communicate between the controller and switches.

pscan's SDN Controller page translates your scan findings into OpenFlow-style flow table entries — giving you a security-lens view of what rules should be applied to your network based on what pscan discovered.

Status Banner

Shows controller health at a glance: active/inactive status, uptime since first scan, controller address (127.0.0.1:6633 — the standard OpenFlow port), and protocol version. The controller is "active" as long as you have scan data in the database.

Metric Cards

MetricWhat it representsWhat it tells you
Connected NodesDistinct hosts scanned at least onceHow many devices are in your scan inventory
Active FlowsTotal open ports across all scansTotal number of potential rule entries needed
Packets ProcessedSimulated counter from scan dataIllustrative only — shows relative traffic volume
Network LoadSimulated % based on flow densityHigh load = many open ports = large attack surface

Flow Table

Each scanned port is translated into an SDN flow entry. The table represents what your OpenFlow switch rules should look like based on scan findings:

ColumnMeaning
Flow IDSequential entry number
Priority300 = DROP rules for dangerous ports · 100 = FORWARD for permitted services. Higher priority = evaluated first.
MatchDestination IP:port and protocol (TCP) that this rule applies to
ActionDROP — block this traffic · FORWARD — allow this traffic to pass
Packets / BytesSimulated counters showing estimated traffic volume
StatusAlert = high-risk port that should be blocked · Active = permitted service

SDN Policy Recommendations (after scan)

After a scan completes on the New Scan page, if high-risk ports were found, pscan automatically generates a list of recommended DROP rules for those ports. You can Copy OpenFlow JSON — this gives you a JSON payload in Ryu/OpenFlow format that can be pushed directly to an OpenFlow switch via a REST API.

High-Risk Ports (automatically flagged)

Connected Nodes Table

Lists every scanned host with scan count, last scan time, and total open ports. Use Re-scan to kick off a new scan directly, or View reports to see the formatted assessment for that host.

Filtering the Flow Table

Type in the search box to filter by IP, port, or service name. Use the All / FORWARD / DROP buttons to filter by action type. This is useful when you want to quickly see all DROP-flagged entries and verify each one.

Tips
  • Filter the flow table to DROP and investigate each flagged port — these are the highest-priority remediation items.
  • Run more scans across your network to populate the controller with a complete picture of your attack surface.
  • Use Ctrl+Shift+R to force-refresh the controller data with the latest scan results.
  • The OpenFlow JSON from the Scan Results page can be pushed to a real Ryu controller's REST API at /stats/flowentry/add.
What is OpenFlow?

OpenFlow is a communication protocol that gives a controller access to the forwarding plane of network switches and routers. When pscan generates a DROP rule JSON, it's in the format that an OpenFlow 1.3-compatible controller (like Ryu or ONOS) can push directly to managed switches — making the security finding instantly actionable at the network layer.

CLI Usage

Run pscan directly from the terminal without the dashboard

Basic Syntax

pscan <target> -p <range> [options]

Examples

# Scan common ports on a local host
pscan 192.168.1.1 -p 1-1024

# Full scan with longer timeout (WAN target)
pscan 10.0.0.5 -p 1-65535 --timeout 3.0

# Specific ports only
pscan myserver.local -p 80,443,8080,8443

# Fewer threads for noisy-network resilience
pscan 192.168.1.1 -p 1-1024 --timeout 2.0 --threads 50

# Disable vulnerability scan for speed
pscan 192.168.1.1 -p 1-65535 --no-vulns

# Add probe delay to reduce IDS noise
pscan 192.168.1.1 -p 1-1024 --delay 0.1

All Flags

FlagDefaultDescription
-p <range>1-1024Port range (1-1024) or comma-separated list (80,443)
--timeout <s>1.0Per-port connection timeout in seconds
--threads <n>100Concurrent scan threads (max 500)
--delay <s>0Probe delay per thread in seconds (0.05–0.2 for low-noise)
--ping-check—Ping host before scanning — abort if unreachable
--aggressive—Fast preset: timeout=0.3s, threads=300 (noisy but quick)
--no-vulns—Skip vulnerability matching (faster port-only scan)

Starting the Dashboard Server

# From inside the Pscan directory:
uvicorn backend.app:app --host 127.0.0.1 --port 8000 --reload

# Or with auto-reload off (production-style):
uvicorn backend.app:app --host 127.0.0.1 --port 8000

Then open http://127.0.0.1:8000 in your browser.

CLI + Dashboard Shared Database

The CLI and the dashboard share the same SQLite file (pscan.db). Scans run from the terminal appear immediately in Scan History when you refresh the browser — and vice versa.

CLI Output

The CLI prints open ports in real time as they are discovered, then saves a .txt report file in the current directory. The file is also linked from Scan History.

Tips
  • The CLI Equivalent box on the New Scan page always shows the terminal command matching your current settings — copy it directly.
  • Use --threads 50 on slow or congested networks to avoid dropped connections.
  • Add --delay 0.1 when scanning a target with IDS — it reduces the burst profile significantly.
  • Run pscan --help for a quick in-terminal reference.
Quick Reference
pscan 192.168.1.1 -p 1-1024
pscan host -p 80,443 --no-vulns
pscan host -p 1-65535 --delay 0.1
pscan host --ping-check
pscan host --aggressive -p 1-65535

FAQ

Common questions and answers

Several possible causes:

  • Firewall — the target has a firewall that silently drops connections. All ports look closed.
  • Wrong IP — double-check the target address. Use ping first to confirm the host is reachable.
  • Narrow port range — try a Full scan (1–65535) to cover every port.
  • Timeout too low — increase timeout to 2–3 s for WAN targets or slow VMs.
  • VM network mode — for VirtualBox/VMware, switch the VM adapter to Bridged so it gets a real LAN IP.

Enable Host Alive Check first — if it reports the host as unreachable, fix connectivity before scanning ports.

Yes. Use the VM's IP address. Network adapter mode matters:

  • Bridged — VM gets a real LAN IP. Most scannable — recommended.
  • NAT — VM gets a private IP from the hypervisor (e.g. 10.0.2.x). Find it with ip addr (Linux) or ipconfig (Windows) inside the VM.
  • Host-only — VM is only reachable from the host machine, not from other LAN devices.

hostname.local also works if mDNS (Bonjour/Avahi) is enabled on the VM and you're on the same LAN segment.

.local hostnames are resolved via mDNS (Multicast DNS) — a zero-configuration protocol that lets devices announce themselves on a LAN without a DNS server.
  • Windows and macOS use Bonjour for mDNS
  • Linux uses Avahi
  • Example: raspberrypi.local resolves to your Pi's IP automatically

Limitation: mDNS only works on the same network segment and does not cross routers. If the device is on a different subnet, use its IP address directly.

pscan uses banner grabbing — connecting to each open port and reading the service's initial response — then matching that banner against a built-in CVE database.

  • Confirmed — a specific vulnerable version string was found in the banner. High confidence, but always verify before patching.
  • Potential — the service type matches a vulnerability category, but the version couldn't be confirmed from the banner. Low confidence — verify manually.

Limitations: services that suppress version banners, use TLS without a version header, or run on non-standard ports may not be detected. pscan is a first-pass triage tool — always follow up with dedicated vulnerability scanners (Nessus, OpenVAS) for production environments.

  • Database: data/database/backend.db — SQLite file containing all scans, ports, and vulnerability records
  • Text reports: data/records/ — one .txt file per scan
  • Exports: data/exports/ — CSV/JSON exports from Scan History
  • Users: data/users.json — registered accounts (passwords hashed with SHA-256)

No data is ever transmitted externally. To back up everything, copy the entire data/ directory. To wipe all scan data, delete data/database/backend.db — it will be recreated empty on next startup.

These modes require raw socket access, which needs Administrator/root privileges and Npcap (Windows) or libpcap (Linux/macOS). A FastAPI web server running as a normal user cannot open raw sockets.

They are available via the CLI when run as Administrator:

# Decoy scan (run as Administrator)
pscan 192.168.1.1 -p 1-1024 -D 5

# MAC spoofing (run as Administrator)
pscan 192.168.1.1 -p 1-1024 -m random --restore-mac

After a scan, pscan makes a heuristic guess at the target OS based on which ports are open. This is not as accurate as active OS fingerprinting (like Nmap's -O flag which analyses TCP/IP stack behaviour) — it's a quick indicator based on service patterns.

OS GuessKey ports that triggered it
Windows3389 (RDP), 135+445 (RPC+SMB), or 139 (NetBIOS)
Linux/Unix22 (SSH) + 80/443, without RDP
macOS22 (SSH) + 548 (AFP)
Printer/Embedded631 (IPP), 9100 (RAW print), or 515 (LPD)
Database Server1433 (MSSQL), 3306 (MySQL), or 5432 (Postgres)

Always treat this as a hint, not a fact — many services run on non-standard ports and the OS guess may be wrong.

Not natively yet, but you can use your OS task scheduler:

# Windows Task Scheduler (run daily at 02:00)
schtasks /create /tn "pscan-daily" /tr "pscan 192.168.1.0/24 -p 1-1024" /sc daily /st 02:00

# Linux cron (run every night at 2am)
0 2 * * * /usr/local/bin/pscan 192.168.1.1 -p 1-1024

Results appear automatically in Scan History the next time you open the dashboard. Scheduled scans and CLI scans share the same database as the dashboard.

  • Add a user: Go to /register (or click "Create account" on the login page) and fill in the form. The new account is stored in data/users.json.
  • Remove a user: Edit data/users.json and delete the relevant entry. Restart the server for the change to take effect.
  • Default admin: The admin / pscan2024 account is hardcoded in backend/config/settings.py. Change AUTH_USERNAME and AUTH_PASSWORD there before exposing pscan on a network.
